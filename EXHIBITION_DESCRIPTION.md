# S-Shield 전시 설명 자료

## 1. 프로젝트 소개

**S-Shield**는 학교폭력 상황을 겪은 학생이 자신의 상황을 차분히 정리하고, 필요한 증거와 대응 방향을 확인할 수 있도록 돕는 AI 기반 상담 및 리포트 서비스입니다.

학교폭력 피해 상황에서는 당사자가 무엇부터 말해야 할지, 어떤 증거를 남겨야 할지, 어디에 도움을 요청해야 할지 혼란스러울 수 있습니다. S-Shield는 사용자의 대화를 바탕으로 사건 내용을 정리하고, 법률 및 학교폭력 대응 절차에 참고할 수 있는 분석 리포트를 제공합니다.

단순히 답변만 제공하는 챗봇이 아니라, 사용자의 말 속에서 사건 내용, 관계, 시점, 증거, 피해 영향, 원하는 도움을 하나씩 확인하며 실제로 활용 가능한 형태로 정리하는 것이 목표입니다.

## 2. 문제의식

학교폭력 피해자는 상황을 설명하는 과정에서 다음과 같은 어려움을 겪을 수 있습니다.

- 사건을 어디서부터 어떻게 설명해야 할지 모름
- 증거를 어떤 방식으로 남겨야 하는지 모름
- 단순 갈등인지 학교폭력 사안인지 판단하기 어려움
- 상담 기록과 신고 자료를 정리하는 데 부담을 느낌
- 감정적으로 불안한 상태에서 혼자 대응하려고 함

S-Shield는 이러한 문제를 줄이기 위해, 사용자가 편하게 대화하면 AI가 필요한 확인 질문을 이어가고, 리포트 형태로 핵심 내용을 정리해 줍니다.

## 3. 주요 기능

### AI 상담

사용자는 채팅 화면에서 자신이 겪은 일을 자연스럽게 입력할 수 있습니다. AI는 입력된 내용을 바탕으로 현재 상황을 이해하고, 필요한 대응 방향을 안내합니다.

예를 들어 SNS 비방, 단체 채팅방 욕설, 신체 폭력, 협박, 따돌림 등 다양한 상황에 대해 증거 보관 방법과 도움 요청 방향을 안내합니다.

### 빠른 답변

상담 중 AI가 추가 확인이 필요한 내용을 빠른 답변 카드로 제공합니다. 사용자는 직접 긴 문장을 작성하지 않아도, 현재 상황에 맞는 답변을 선택해 상담을 이어갈 수 있습니다.

빠른 답변은 단순 선택지가 아니라, 리포트에 필요한 핵심 정보를 자연스럽게 모으기 위한 구조입니다.

### 리포트 생성

상담을 통해 충분한 정보가 모이면, 사용자는 분석 리포트를 생성할 수 있습니다. 리포트에는 다음과 같은 내용이 포함됩니다.

- 사건 요약
- 폭력 유형
- 위험도 점수
- 판단 근거
- 우선 대응 조치
- 관련 법령
- 증거 정리 가이드

리포트는 상담 내용을 바탕으로 자동 구성되며, 사용자가 보호자, 학교, 상담 선생님 등에게 상황을 설명할 때 참고 자료로 활용할 수 있습니다.

### 상담 및 리포트 기록 관리

로그인한 사용자는 마이페이지에서 상담 기록과 리포트 기록을 확인할 수 있습니다. 각 상담 세션에 연결된 리포트를 다시 열 수 있고, 필요 없는 기록은 삭제할 수 있습니다.

상담 세션을 삭제하면 연결된 대화와 리포트도 함께 정리되어, 사용자가 자신의 기록을 직접 관리할 수 있습니다.

## 4. 사용자 흐름

1. 사용자가 학교폭력 관련 상황을 채팅으로 입력합니다.
2. AI가 상황을 분석하고 필요한 확인 질문을 제시합니다.
3. 사용자는 빠른 답변 또는 직접 입력으로 추가 정보를 제공합니다.
4. 사건 내용, 관계, 시점, 증거, 피해 영향 등이 충분히 모이면 리포트 카드가 표시됩니다.
5. 사용자는 리포트를 열어 사건 요약과 대응 방향을 확인합니다.
6. 필요하면 상담을 이어가거나, 마이페이지에서 기록을 다시 확인합니다.

## 5. 기술적 특징

S-Shield는 프론트엔드와 백엔드가 분리된 웹 서비스 구조로 제작되었습니다.

- 프론트엔드: React 기반 사용자 인터페이스
- 백엔드: Spring Boot 기반 API 서버
- 데이터 관리: 상담 세션, 메시지, 리포트 저장
- 배포: 프론트엔드와 백엔드를 각각 클라우드 환경에 배포
- AI 분석: 상담 내용을 바탕으로 학교폭력 가능성, 증거 필요성, 대응 방향 분석

특히 상담 기록과 리포트가 섞이지 않도록 세션 단위로 관리하며, 기존 리포트 이후 다른 사건이 입력되면 기존 리포트에 무리하게 합치지 않도록 처리했습니다.

## 6. 화면별 개발 및 작동 흐름

이 부분은 전시장에서 “이 서비스가 실제로 어떻게 동작하나요?”라는 질문을 받았을 때 설명하기 좋은 기술 흐름입니다.

### 전체 구조

S-Shield는 크게 **프론트엔드 화면**, **백엔드 API**, **데이터베이스**, **AI 분석 로직**으로 나뉩니다.

```text
사용자 화면
  ↓
React 프론트엔드
  ↓
Spring Boot API 서버
  ↓
서비스 로직(ChatService, ReportService, AnalysisService)
  ↓
DB 저장(Session, Message, Report, User)
  ↓
AI 분석 및 리포트 생성
```

프론트엔드는 사용자가 보는 화면과 버튼 동작을 담당하고, 백엔드는 상담 내용 저장, 로그인 확인, 리포트 생성, 리포트 조회, 세션 관리 같은 핵심 로직을 담당합니다.

### 메인 화면

메인 화면은 사용자가 서비스를 처음 접하는 진입 화면입니다.

작동 흐름:

1. 사용자가 `시작하기` 버튼을 누릅니다.
2. 프론트엔드 라우터가 `/chat?new=1` 경로로 이동시킵니다.
3. 채팅 화면은 새 상담 상태로 초기화됩니다.
4. 로그인하지 않은 사용자는 임시 상담으로 시작하고, 로그인한 사용자는 상담 기록 저장이 가능한 상태로 진입합니다.

개발 포인트:

- React Router로 화면 이동을 처리했습니다.
- 메인 화면은 기능 설명보다 상담 시작 진입에 집중하도록 구성했습니다.
- 사용자가 바로 서비스를 체험할 수 있도록 첫 화면에서 채팅으로 연결됩니다.

### 로그인 화면

로그인 화면은 상담 기록과 리포트 저장 기능을 사용하기 위한 인증 단계입니다.

작동 흐름:

1. 사용자가 이메일/비밀번호로 로그인하거나 OAuth 로그인을 진행합니다.
2. 백엔드 `AuthController`가 사용자 정보를 확인합니다.
3. 인증에 성공하면 JWT 토큰이 발급됩니다.
4. 프론트엔드는 토큰을 브라우저 localStorage에 저장합니다.
5. 이후 API 요청 시 Authorization 헤더에 토큰을 포함합니다.

개발 포인트:

- JWT 기반 인증 구조를 사용했습니다.
- 로그인 사용자는 상담 세션과 리포트를 DB에 저장할 수 있습니다.
- 비로그인 사용자는 임시 상담은 가능하지만, 기록 저장 기능은 제한됩니다.

### 채팅 화면

채팅 화면은 S-Shield의 핵심 화면입니다. 사용자가 상황을 입력하면 AI가 상담을 이어가고, 필요한 경우 빠른 답변과 리포트 CTA를 표시합니다.

작동 흐름:

1. 사용자가 메시지를 입력합니다.
2. 프론트엔드가 `/chat/message` API로 메시지를 전송합니다.
3. 백엔드 `ChatController`가 요청을 받고 `ChatService`로 전달합니다.
4. `ChatService`는 상담 세션을 확인하거나 새 세션을 생성합니다.
5. 사용자 메시지를 `Message` 테이블에 저장합니다.
6. 지금까지의 대화 기록을 바탕으로 AI 응답을 생성합니다.
7. AI 응답도 다시 `Message` 테이블에 저장합니다.
8. 프론트엔드는 응답을 받아 채팅창에 표시합니다.

개발 포인트:

- 상담은 `Session` 단위로 관리됩니다.
- 각 메시지는 `Message`로 저장되며, 역할은 `user`와 `assistant`로 구분됩니다.
- 기존 세션을 다시 열면 해당 세션의 메시지와 연결된 리포트 상태를 함께 불러옵니다.
- 모바일에서 버튼이나 빠른 답변이 잘리지 않도록 반응형 UI를 조정했습니다.

### 빠른 답변 흐름

빠른 답변은 사용자가 긴 문장을 직접 쓰지 않아도 리포트에 필요한 정보를 제공할 수 있도록 만든 기능입니다.

작동 흐름:

1. 백엔드가 현재 대화에서 부족한 정보를 판단합니다.
2. 부족한 축이 있으면 확인 질문과 선택지를 생성합니다.
3. 프론트엔드는 AI 메시지 아래에 빠른 답변 카드로 표시합니다.
4. 사용자가 선택지를 누르면 자연스러운 사용자 문장으로 채팅에 전송됩니다.
5. 해당 답변은 일반 메시지처럼 DB에 저장되고, 이후 리포트 판단에 반영됩니다.

개발 포인트:

- 빠른 답변은 단순 UI가 아니라 리포트 생성 조건을 채우는 데이터 수집 장치입니다.
- 같은 질문을 반복하지 않도록 이미 답변한 항목을 히스토리에서 확인합니다.
- 저장된 상담을 다시 열었을 때도 최신 빠른 답변 카드가 자동으로 펼쳐지도록 처리했습니다.

### 리포트 생성 흐름

리포트는 사용자의 상담 내용을 바탕으로 사건을 구조화해 보여주는 결과물입니다.

작동 흐름:

1. 상담 내용이 충분히 모이면 프론트엔드에 리포트 CTA가 표시됩니다.
2. 사용자가 `리포트 열기` 또는 `상담 내용 정리하기` 버튼을 누릅니다.
3. 프론트엔드가 `/reports/generate` API를 호출합니다.
4. 백엔드 `ReportController`가 요청을 받고 `ReportService`로 전달합니다.
5. `ReportService`는 해당 세션의 메시지를 모아 분석합니다.
6. `AnalysisService`가 폭력 유형, 위험도, 판단 근거, 대응 조치, 증거 가이드를 계산합니다.
7. 결과는 `Report` 테이블에 저장됩니다.
8. 프론트엔드는 `/result` 화면으로 이동해 리포트를 보여줍니다.

개발 포인트:

- 리포트는 상담 세션과 연결되어 저장됩니다.
- 리포트가 이미 있는 세션에서 추가 대화가 생기면 `리포트 갱신하기` 상태로 바뀝니다.
- 단, 기존 리포트와 다른 사건으로 보이는 내용은 기존 리포트에 섞지 않고 차단합니다.
- 이를 통해 여러 사건이 하나의 리포트에 섞이는 문제를 줄였습니다.

### 리포트 결과 화면

리포트 결과 화면은 상담 내용을 실제로 제출하거나 설명할 때 참고할 수 있도록 정리된 화면입니다.

작동 흐름:

1. 사용자가 리포트 화면에 접근합니다.
2. 프론트엔드가 URL의 report id 또는 session id를 확인합니다.
3. 백엔드에서 해당 리포트를 조회합니다.
4. 로그인 사용자인 경우 리포트 소유자와 현재 사용자가 일치하는지 확인합니다.
5. 권한이 맞으면 리포트 데이터를 화면에 표시합니다.

리포트 화면에 표시되는 정보:

- 사건 요약
- 위험도 점수
- 폭력 유형
- 판단 근거
- 우선 대응 조치
- 관련 법령
- 증거 정리 가이드

개발 포인트:

- 다른 사용자의 리포트에 접근하지 못하도록 백엔드에서 소유자 검사를 합니다.
- 리포트 화면은 자동 이동이 아니라 사용자가 버튼을 눌러 확인하는 방식으로 구성했습니다.
- 위험도와 조치 내용이 한 화면에서 잘 보이도록 카드형 레이아웃으로 정리했습니다.

### 마이페이지 화면

마이페이지는 로그인한 사용자가 자신의 상담 기록과 리포트 기록을 관리하는 화면입니다.

작동 흐름:

1. 사용자가 마이페이지에 접근합니다.
2. 프론트엔드는 `/users/stats`, `/reports`, `/chat/sessions` API를 호출합니다.
3. 백엔드는 현재 로그인한 사용자 기준으로 데이터만 조회합니다.
4. 프론트엔드는 상담 기록과 리포트 기록을 리스트 형태로 표시합니다.
5. 사용자는 리포트를 열거나 상담 세션을 다시 열 수 있습니다.
6. 필요 없는 상담 또는 리포트는 삭제할 수 있습니다.

개발 포인트:

- 상담 기록과 리포트 기록을 한 화면에서 관리할 수 있도록 구성했습니다.
- 리스트는 박스 내부 스크롤 형태로 만들어 화면이 너무 길어지지 않도록 했습니다.
- 상담 세션을 삭제하면 연결된 대화와 리포트도 함께 정리되도록 설계했습니다.
- 리포트 카드에서는 제목, 날짜, 폭력 유형, 위험도 점수가 한눈에 보이도록 배치했습니다.

### 세션 관리 흐름

S-Shield는 모든 상담을 세션 단위로 관리합니다.

작동 흐름:

1. 사용자가 새 상담을 시작하면 새로운 `Session`이 생성됩니다.
2. 사용자의 메시지와 AI 응답은 해당 세션에 연결되어 저장됩니다.
3. 리포트도 해당 세션에 연결됩니다.
4. 사용자가 과거 상담을 열면 해당 세션의 메시지와 리포트를 다시 불러옵니다.

개발 포인트:

- 세션 단위 관리 덕분에 상담과 리포트가 서로 섞이지 않습니다.
- 리포트 메뉴에서 들어갈 때도 가능한 경우 현재 세션에 연결된 리포트를 우선 표시합니다.
- 세션 id로 접근하더라도 백엔드에서 소유자 검사를 하기 때문에 다른 사용자의 기록을 볼 수 없습니다.

### 백엔드 분석 흐름

백엔드 분석은 크게 세 단계로 나뉩니다.

```text
대화 기록 수집
  ↓
리포트 준비도 판단
  ↓
폭력 유형·위험도·증거·조치 분석
```

주요 역할:

- `ChatService`: 상담 흐름, AI 응답, 빠른 답변, 리포트 준비 상태 관리
- `ReportService`: 리포트 생성, 조회, 갱신, 세션별 리포트 관리
- `AnalysisService`: 폭력 유형, 위험도, 관련 법령, 판단 근거 분석
- `EvidenceGuideService`: 상황에 맞는 증거 정리 항목 생성
- `LawDataService`, `LawApiService`: 관련 법령 데이터 제공

개발 포인트:

- 단순히 AI 답변만 보여주는 것이 아니라, 백엔드에서 리포트 생성 가능 조건을 별도로 판단합니다.
- 사건 내용, 관계, 시점, 증거, 피해 영향, 원하는 도움 등이 모였는지 확인합니다.
- 리포트 이후 추가 대화가 들어오면 같은 사건 보강인지, 다른 사건인지 검사합니다.
- 다른 사건이면 기존 리포트에 섞지 않고 새 상담으로 분리하도록 유도합니다.

### 전시에서 말하기 좋은 기술 요약

S-Shield는 React 화면에서 사용자 입력을 받고, Spring Boot 백엔드가 상담 세션과 메시지를 저장한 뒤, AI 분석 로직을 통해 학교폭력 유형과 위험도, 증거 가이드, 대응 조치를 리포트로 구조화하는 서비스입니다. 특히 상담과 리포트를 세션 단위로 연결하고, 다른 사건이 기존 리포트에 섞이지 않도록 막는 점이 핵심 기술 포인트입니다.

## 7. 안전성과 윤리적 고려

S-Shield는 신고를 대신 접수하거나 법적 판단을 확정하는 서비스가 아닙니다. AI 분석 결과는 참고용이며, 실제 신고나 조치는 보호자, 학교 담당자, 상담 기관, 경찰 등 신뢰할 수 있는 기관과 함께 진행해야 합니다.

또한 생명이나 신체에 대한 위협, 보복 우려, 즉각적인 위험이 있는 경우에는 AI 상담보다 먼저 보호자, 학교 담당자, 117 또는 112에 연결하도록 안내합니다.

서비스는 사용자가 혼자 판단하도록 만들기보다, 상황을 정리하고 도움을 요청할 수 있도록 돕는 보조 도구로 설계되었습니다.

## 8. 기대 효과

S-Shield를 통해 사용자는 다음과 같은 도움을 받을 수 있습니다.

- 자신의 상황을 차분히 정리할 수 있음
- 필요한 증거를 놓치지 않고 확인할 수 있음
- 학교나 보호자에게 설명할 자료를 준비할 수 있음
- 상담 기록과 리포트를 한 곳에서 관리할 수 있음
- 학교폭력 대응 과정에서 느끼는 막막함을 줄일 수 있음

## 9. 전시 설명용 한 문장

S-Shield는 학교폭력 피해자가 자신의 상황을 AI와 대화하며 정리하고, 필요한 증거와 대응 방향을 리포트로 확인할 수 있도록 돕는 상담 지원 서비스입니다.

## 10. 발표 마무리 멘트

학교폭력 상황에서 가장 어려운 점은 피해자가 혼자서 모든 것을 정리하고 설명해야 한다는 부담입니다. S-Shield는 그 부담을 줄이고, 사용자가 자신의 상황을 더 안전하고 명확하게 전달할 수 있도록 돕는 것을 목표로 합니다.

AI가 모든 문제를 해결할 수는 없지만, 사용자가 도움을 요청하기 전까지의 막막한 과정을 조금 더 쉽게 만들어 줄 수 있습니다.
